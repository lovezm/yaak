# Lessons

- 当用户希望头信息、键值列表更便于复制或更贴近原始文本时，优先使用 `KEY:VALUE` 单行展示，不要默认拆成左右两列。
- 当用户明确说“方便复制”时，不要只把视觉样式改成单行；优先输出为整段纯文本块，避免 `span` 或多元素拼装影响整段选中。
- 当用户要“生成代码”时，优先把结果直接嵌在当前面板可见区域，并提供一键复制，不要额外增加跳转或弹窗步骤。
- 当通过弹窗批量修改当前面板数据时，要确认当前视图会立即刷新；如果内部组件有本地状态，需要在保存后显式触发重建，避免“数据已保存但界面要重载才显示”。
- 如果保存动作依赖异步 `patchModel` 回流，不要在回流前就用旧 props 强制重建组件；优先在中间层维护一个本地即时态，先把用户刚改的内容显示出来。
- 请求头编辑器里的字段展示要贴近真实调试需求：像 `cookie` 这种用户明确要求可见的头值，不要默认按密码字段隐藏。
- 界面文案不要机械地全量中文化；像 `Params` 这类用户明确指定要保留英文的术语，应按用户习惯直接使用英文标签。
- 当参数面板和 URL 输入框表达的是同一份请求地址信息时，优先做即时双向联动，不要让用户改完 `Params` 还要自己 mentally 同步 URL。
- 当用户要求“本次请求使用单独代理”这类请求级网络能力时，不能只加输入框；必须同时打通模型持久化、模板渲染和真实发送链路，否则界面看起来可填但请求不会真正走代理。
- 当网络层引入“按请求变化”的配置时，必须检查 HTTP 客户端缓存键是否包含这些配置；如果缓存只按上下文 id 复用，代理、证书、DNS 覆盖这类变更都会看起来“已保存但未生效”。
- 当用户已经明确给出自动判断规则时，要按用户的话直接实现，不要继续做“更聪明”的推断；这次请求级代理里，`http/https` 输入就应一律按代理 API 处理，而不是再试图兼容成直连代理 URL。
- 当用户说明“默认是 HTTP 代理”时，底层构造也要贴这个语义；同一条代理地址不应机械拆成 http/https 两套规则，优先收敛成一条 shared proxy 规则。
- 当用户直接给出某种语言的代码生成模板时，优先贴着模板生成，不要擅自抽象成“更通用”的写法；尤其是像易语言这类工具型代码，变量名、调用顺序和字符串拼接风格都应尽量对齐用户示例。
- 当用户给出代码生成的细粒度格式纠正时，要严格按该语言的实际书写习惯收口；例如易语言 `GET` 模板里不需要额外生成 `局_提交数据 = ""` 这种无意义语句。
- 当用户对某种语言模板的“参数位是否留空”提出纠正时，要连同调用参数一起同步调整；例如易语言 `GET` 模板里不仅不需要赋值 `局_提交数据`，调用 `网页_访问_对象(...)` 时对应提交数据参数也应留空。
- 当用户明确要求“某方法一律不要生成某段代码”时，要以方法为最高优先级覆盖正文内容本身；例如易语言 `GET` 代码生成里，即便原请求误带了 body，也不能再生成任何 `局_提交数据` 相关赋值。
- 当用户要 GitHub Actions “手动触发”时，不要默认再附带 `push` 自动触发；发布类工作流应优先只保留 `workflow_dispatch`，避免用户一推送就开始打包。
- 当用户要求某一块界面继续汉化时，不要只改主面板；像侧边栏新增菜单、右键菜单这类二级入口也要沿着同一交互链一起补齐文案。
- 当用户要求“代码生成不用发送请求也能生成”时，不能继续依赖响应快照或发送记录；应直接基于当前请求编辑态实时生成，发送结果只应作为补充而不是前提。
- 当用户指出某个下拉菜单或历史菜单还没汉化时，要连同分组标题、按钮提示和带数量的动态文案一起补齐，不能只翻静态按钮。
- 当用户指出首页或空状态页没汉化时，要继续往热键标签、空状态按钮这种共享显示层追，不要只改页面按钮文本。
- 当用户要求某个网络行为在“当前请求”里单独可控时，不要停留在工作区设置；像禁止重定向这类开关要做成请求级字段并真正进入发送配置。
- 当用户明确要求某个请求级能力“单独可见”时，不要把它塞进别的标签页里；像禁止重定向这种高频开关应给独立入口，而不是藏在代理页内部。
- 当用户明确说“不需要自动检查”时，要同步检查仓库里的 `ci` 工作流触发器，优先把 `push` / `pull_request` 改成仅 `workflow_dispatch`。
- 当请求级开关已经影响真实发送行为时，代码生成模板也要同步反映出来；例如易语言模板里勾选“禁止重定向”后，应在 `网页_访问_对象(...)` 对应参数位生成 `真`。
- 当某个面板需要跟随当前请求编辑态实时更新时，不要在组件内部用一次性的 `getModel()` 拿静态快照；应传入当前请求对象或订阅式数据源，否则就会出现“必须发送或刷新后才显示”的假象。
- 当用户明确指出“下拉切换不好用”时，像请求正文类型这种高频主流程操作应优先改成可直点的标签式切换，不要把核心选项藏在二级下拉菜单里。
- 当用户指出顶部或二级菜单仍有英文时，不要只改当前可见一项；要沿着同一菜单链路把主菜单、子菜单、弹窗标题一起补齐，避免“点进去还是英文”。
- 当用户指出同步、拉取或确认弹窗仍有英文时，不要只翻弹窗标题；表头、说明文案、操作词和按钮也要一起汉化，否则视觉上仍会明显割裂。
- 当用户指出底部 Git 菜单还有英文时，要把主菜单、分支子菜单、远端管理和历史弹窗一起检查；Git 入口通常是一整条交互链，零散翻译会很快露馅。
- 当用户指出某个设置入口还没汉化时，要把入口按钮、弹窗标题、不同状态下的说明、确认提示和辅助按钮一起检查；像加密这种流程型设置，文案往往散在多个状态分支里。
- 当用户把上游项目 fork 成自己的发行版时，不要只改界面文案；设置菜单、更新跳转、反馈入口这类外链也要同步改成 fork 自己的地址，避免还跳回上游官网。
- 当用户指出设置子页里“还有英文列表项”时，要继续追到共享定义层；像快捷键页的范围名和动作名通常不在页面组件里，而在统一的标签映射文件中。
- 当用户明确要求新增某个设置页时，不能只把组件文件写出来；必须确认它已经真正接进 `Settings.tsx` 的标签数组、标签文案和内容渲染，否则界面上根本看不到。
- 当用户要求某个功能“在另一个菜单里也要有入口”时，不能只确认页面本身存在；还要把对应的快捷入口补到用户实际常点的下拉菜单或工具栏里。
- 当用户对“关于”这类信息页继续补充展示项时，要沿用同一信息分组直接补齐，不要另起新的交互入口或弹窗。
- 当用户要求某个教程链接既出现在提示入口里又出现在设置页里时，要把整条使用路径一起补齐；不要只加一个外链按钮，忽略真正操作页里的对应位置。
- 当 fork 版应用要改成自己的更新来源时，不能只改前端“检查更新”按钮；必须同时切掉旧的上游自动 updater 触发点，否则会出现两套更新提示互相打架。
